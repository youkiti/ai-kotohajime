## 1. 背景と目的

### 1.1 背景
- 日本の医療現場で生成AIの活用が広がる一方、学習者(研修医・学生等)に使い方を身につけさせる標準的な方法がない。
- 緩和ケア講習会やICLSのような指導者養成型モデルは、教える側全員に大きな負担を課す。指導者に「教える仕事」を新たに発生させる設計そのものを避ける。

### 1.2 目的
医療系教育者(medical teacher)が**「ここを見ておいて」とリンクを送るだけ**で、教え子が生成AIの実践的な使い方を自習で身につけ、フィードバックをAIから受け取り、**修了証まで自分で発行できる**Webサイトを提供する。

### 1.3 設計原則
1. **教員の手間を増やさない。** 教員の仕事は「リンクを送る」のみ。
2. 学習者は体験、フィードバック、修了証発行まで一人で完結できる。
3. 保守は運営者一人がMarkdown編集だけで回る。指示文の正本もリポジトリ内に置く(v0.8以降、指示文=学習者が実際にコピーして使う採点プロンプトそのものであり、外部サービスへの同期作業は存在しない)。
4. **ユーザーのログ・個人情報をサイト側に一切残さない。**(クイズ合格状態・実践課題提出日時のlocalStorage保存は端末内で完結するため本原則に抵触しない。5.2参照)

## 2. 対象者

- **一次利用者(コンテンツの読者): 学習者** — 研修医、医学生、看護学生、若手医療者など
- **二次利用者(割り当てる人): 医療系教育者** — 指導医、看護教員等。FDの対象はこちら
- 前提スキル: Git不要。ブラウザとコピペ操作のみ。ただし**手元で使える生成AIツール(ChatGPT、Claude、Gemini、Copilot等。無料版で可)を1つ持っていること**を前提とする(各ユニットの演習とAIフィードバックの両方で使用する)

## 3. 非目標(やらないこと)

- 認定制度(第三者検証を伴う公式な資格・修了認定)
- ログイン・進捗管理・LMS機能・アクセスログの収集
- API連携によるページ内チャット
- 教員向けの授業台本・講義スライド
- 指導者養成コース
- 特定ツールへの必須依存(採点プロンプトは任意の生成AIで機能する文面とする)

## 4. コンテンツ要件

### 4.1 学習者向け自習ユニット(本体)

- 1ユニット10分以内、症例ベースで臨床的に切実なお題
- 構成要素:
  1. なぜこのスキルが必要か(1段落)
  2. コピー可能なプロンプト(ツール非依存)
  3. 手元の生成AIツールに貼って実行する手順
  4. 期待される出力の例
  5. うまくいかない典型パターンと対処
  6. 安全上の注意(患者情報を入れない等)
  7. 自己チェッククイズ(静的JS、ブラウザ内完結)
  8. 成果物フィードバック: 演習を行った対話履歴の最後に採点プロンプト(4.2)を送り、AIから講評を受ける
  9. 修了証の発行(下記4.3)
  10. 参考資料(任意): 外部公開資料へのリンク。正本は `documents/AI事始-参考資料-*.md` で一括管理。リンクのみで転載せず、参照しなくてもユニット単体で完結する構成を崩さない(クイズ出題範囲に含めない)

### 4.2 AIフィードバックの仕組み

**提供形態(単一方式・v0.8で確定)**

- フィードバック用の**採点プロンプト**(ルーブリック込み)をサイトに掲載し、学習者が手元の任意のAI(ChatGPT/Claude/Gemini/Copilot等)で使う。**このプロンプトが指示文の正本**(`docs/feedback.md`)であり、学習者が実際にコピーして使うものそのもの
- **使い方(基本)**: 演習を行った**対話履歴の最後に採点プロンプトを送信**する。AIが履歴全体(反復改善の過程を含む)を踏まえて講評できる
- **使い方(フォールバック)**: 対話履歴が残っていない場合や別ツールで講評を受けたい場合は、新しいチャットに採点プロンプトと成果物(自分が使ったプロンプトとAIの出力のコピー)をまとめて貼る
- v0.7までの**Gem版は廃止**(v0.8)。理由: 指示文の手動同期という保守作業をなくすため。また演習自体が手元の生成AIを必要とするため、「Googleアカウント不要で使える」というGem固有の利点は実質的な障壁低減になっていなかった

**採点プロンプトの指示文要件**
- 課題(ユニット)ごとのルーブリックに基づく講評(お世辞防止)
- 出力形式を固定: **1行目は固定見出し「【AI事始 講評】」**(領域B実践課題の提出検証マーカーを兼ねる。4.6参照)+ 良かった点1つ + 改善点2つ + 改善版プロンプトの提案
- 講評対象(対話履歴全体、または貼り付けられた成果物)に個人情報・患者情報らしき記載を検知したら、講評より先に指摘して中断

### 4.3 修了証(ログレス・自己発行型)

**仕組み**
- **発行単位は領域ごと**。呼称は「目録」(例:「A領域 目録」)。初級3領域すべての目録取得で「初級目録」を発行(将来の上級で「免許皆伝」を構想)
- **発行条件(v0.8で改訂)**:
    - 領域A・C: 領域クイズの合格(正答率8割以上、再挑戦無制限)のみ
    - **領域B: 領域クイズの合格 + 実践課題の提出**(4.6)。両方がそろった時点で発行ボタンを表示する。達成の順序は問わない
    - いずれの領域も**ユニット本文の閲覧は要件としない**(既修者はクイズ・課題に直行して修了可能)
- 学習者が氏名を入力→ブラウザ内(JavaScript/Canvas)で**PNG**を生成→ダウンロード(jsPDFは日本語フォント同梱が重いため不採用)
- 氏名欄は**自由入力(ニックネーム可)**。SNS等に実名を出したくない場合はニックネームで発行し直せる旨をUIに明記
- **SNS共有(2段構え)**: (a) Web Share API対応環境(主にモバイル)では生成PNGをOSの共有シート経由で直接共有、(b) 非対応環境ではPNGダウンロード+文面プリセット済みのXポスト画面を開き画像は手動添付。個別画像のOGPカードはサーバに画像を置く必要があるため不採用(ログレス原則)
- **サーバ送信ゼロ。氏名・成績・アクセス記録はどこにも保存されない**(GitHub Pagesにはバックエンドが存在しないため構造的に担保)
- 記載事項: 氏名、ユニット/領域名、発行日、サイト名、免責文言。領域Bの目録本文は実践課題の修了も反映した文言とする

**位置づけ(重要)**
- 検証機構を持たない自己申告型であり、**公式な研修証憑ではない**ことを修了証面とサイトに明記
- 用途: (a) 学習の動機づけ・達成感、(b) 教員への任意の修了報告(学習者が修了証画像を指導医に送る運用を教員向けガイドで案内)

### 4.4 教員向け割り当てガイド(薄い層)

- 課題指示文テンプレート(Slack/メールにコピペするだけの文面。ユニットURL入り)
- 修了証画像の提出を求める場合の文例
- 任意: 15分振り返り会用の問いリスト

### 4.5 コンテンツ領域と優先順位

コンテンツは初級コンピテンシー(別文書「AI事始-初級コンピテンシー」参照)に基づいて構成する。

| 優先 | 領域 | 備考 |
|---|---|---|
| MVP(初級) | A: 生成AIのメカニズム | LLM/拡散モデルの原理、ハルシネーションの由来、マルチモーダル。2ユニット |
| MVP(初級) | B: 対話的活用スキル | 反復改善、コンテキスト付与、AIに質問・批評させる技法。2ユニット。題材は職種別(医師/看護/薬学)タブで差し替え、構造・クイズは共通。**目録発行にはクイズ合格に加えて実践課題(4.6)の提出が必要** |
| MVP(初級) | C: 倫理・安全・検証 | 患者情報・個人情報、著作権、バイアス、人による検証。2ユニット |
| Phase 2 | 研究(文献検索、抄読会準備) | |
| Phase 3 | 画像生成の実践 | 講義スライド用図解に限定。解剖学的誤り・著作権の注意を必ず併記(仕組み自体は初級A領域で扱う) |
| 今後の検討 | 中級・上級 | 初級の反響を見てから設計 |

### 4.6 クイズ・実践課題仕様(確定)

**領域クイズ**

- 四択・一括採点(全問回答後に採点)。正答率8割以上で合格。不正解の設問には解説を表示
- 問題数(v0.8で改訂): **領域A・C = 10問(8問以上で合格)、領域B = 5問(4問以上で合格)**。領域Bはシナリオ型設問(状況を提示し、次に取るべき行動を選ばせる)で構成し、実践課題と組み合わせて評価するため問題数を絞る
- 再挑戦: 無制限(設問プールからのランダム出題は将来検討)
- 合格状態は**localStorageに保存**(端末内のみ。キーは領域単位)。3領域の修了検知により「初級目録」発行ボタンを有効化
- ユニット内の自己チェッククイズは同じエンジンを使うが合否・保存に関与しない(理解確認のみ)

**領域B実践課題(v0.8で新設)**

- 学習者はUnit B-1またはUnit B-2のお題(職種は問わない)で演習を行い、その**対話履歴の最後に採点プロンプト(4.2)を送信**して講評を受け、返ってきた**講評テキストを領域Bクイズページの提出欄に貼り付けて提出**する
- サイト側の検証は「固定見出し**【AI事始 講評】**を含む」「最低文字数以上」の**テキスト存在確認のみ**。講評の内容や成績は判定しない(採点=形成的フィードバックはAIが担い、サイトは実施の記録のみを行う)
- 貼り付けられたテキストは**保存も送信もしない**。localStorageに記録するのは提出日時のみ(キーは領域単位)
- 既知の限界: 固定見出しは採点プロンプト自体にも記載されているため、講評以外のテキスト(プロンプト本文など)を貼っても検証を通過しうる。これは目録が検証機構を持たない自己申告型である(4.3)ことから**許容する**(静的サイトでは原理的に防止不能であり、対策は費用対効果に見合わない)

### 4.7 陳腐化対策
- 普遍的内容とツール固有How-toを分離し、更新対象を局所化
- 採点プロンプトは特定ツールに依存しない文面とし、任意の生成AIで機能するよう維持する
- 各ページに最終更新日を自動表示
- 外部参考資料はリンクのみ掲載し、リンク確認日を参考資料文書(正本)に記録・改訂時に全リンク再確認(別文書「AI事始-参考資料-JSME-ICTシンポジウム」参照)

## 5. 技術要件

### 5.1 構成
- ホスティング: GitHub Pages
- サイトジェネレータ: **Material for MkDocs(確定)**
- 自己チェッククイズ・領域クイズ: 静的JS(ブラウザ内完結、設問データはリポジトリ内JSON)
- 領域B実践課題の提出確認: 静的JS(固定見出し+最低文字数のテキスト存在確認のみ。貼付テキストは保存しない)
- 修了証生成: 静的JS + Canvas → PNG。共有はWeb Share API+フォールバック。外部通信なし
- フィードバックAI: 採点プロンプト(正本 `docs/feedback.md`。学習者が手元の生成AIにコピペして使用)

### 5.2 保守性・プライバシー要件
- 運営者がMarkdown編集のみで更新完結(指示文含む)。クイズ設問はJSONファイル編集(テキストエディタで完結)
- push→自動ビルド・公開
- サイト本体の外部サービス依存ゼロ。フィードバックも学習者の手元の生成AIで完結するため、サイトが依存・運用する外部サービスはない
- アクセス解析を含む一切のユーザーログを収集しない(設計原則4)
- localStorageに保存するのは**クイズ合格日時と実践課題の提出日時のみ**(いずれも学習者の端末内で完結し、サーバには一切送信されない)。実践課題で貼り付けられた講評テキスト自体は**端末内にも保存しない**。この旨をプライバシー方針に明記する
- 既知の許容事項(2026-07-13監査で確認): Material for MkDocsのバンドルには、ResizeObserver非対応の旧式ブラウザに限り互換ポリフィルを外部CDN(unpkg)から読み込む条件分岐が含まれる。解析目的の通信ではなく、想定読者の一般的なブラウザでは発生しない。プライバシー方針には「サイト独自コードは外部送信しない」と限定表現で明記済み

### 5.3 コントリビューション
- PRは受け付けるが主要経路とはみなさない
- GitHub Issueを誤り報告・要望の窓口として案内
- PRレビュー基準(医学的正確性、個人情報記載の有無)をCONTRIBUTING.mdに明文化
